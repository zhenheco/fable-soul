# Fable Soul

[English](README.md) | 繁體中文

**AI coding agent 的判斷力層（judgment layer）— 讓你的 AI 像資深工程師一樣思考、驗證、溝通。**

MIT 授權 · 支援 Claude Code 與 Codex · 純 Markdown + 兩個無相依的小 Python 腳本

---

## 要解決的問題

大多數 AI agent 的失敗不是知識不足。模型明明會寫那段程式碼，缺的是*判斷力*——資深工程師不假思索就會做的操作紀律：

| Agent 常見行為 | 資深工程師的做法 |
|----------------|------------------|
| 改完檔案就說 **「搞定！bug 修好了」**——什麼都沒跑過 | 跑測試、貼輸出，*然後*才說完成 |
| 你說調 timeout 它就調，心裡明知沒用 | 告訴你 timeout 不是原因，然後找出真正的原因 |
| **「現在應該快很多了」** | 「3.4 秒 → 0.06 秒，實測」 |
| 把症狀蓋掉讓錯誤訊息消失 | 一句話說清 bug 的機制，修*那個機制* |
| 診斷出真正的 bug，然後問 **「要不要我修？」** | 直接修——範圍內又可逆的事不用問 |
| 把出乎意料的測試結果解釋掉，維持故事的完整 | 停下來深挖——意外之處才是真 bug 的所在 |
| 一行的問題回一份五個段落的報告 | 一句話回答 |

Fable Soul 把右欄打包成可載入的規則。每一條規則都是從**真實的 agent 失敗**中捕捉、為防止那個特定失敗而寫、並經過測試確認真的會改變行為。它不是「最佳實踐」式的空話合集——它是操作基礎設施。

## 內容結構

```
fable-soul/
├── SKILL.md                        # Skill 進入點（三種模式：Embody / Maintain / Transfer）
├── references/
│   ├── soul.md                     # 判斷力規則本體 — 本專案的核心
│   ├── maintenance.md              # Mirror 地圖、同步流程、失敗捕捉迴圈
│   └── transfer-prompts.md         # 6 個把經驗轉成 skill 的 prompt 模板
└── scripts/
    ├── sync_soul.py                # 同步規則到所有安裝位置、偵測 drift
    └── validate_skill.py           # Skill 套件結構驗證
```

## 模組一 — 判斷力規則（`references/soul.md`）

核心檔案。約 250 行，每個 session 都載入。分三層：

### Operating Gates（操作關卡）

可執行的檢查點，不是感覺：

- **Mode Selector** — 簡單問題直接回答；執行型任務走完整關卡；高風險/面向客戶的工作加強證據要求。不需要儀式的事就不上儀式。
- **Task Start Gate** — 動手前必須能回答三件事：使用者真正要的結果是什麼（*目標*）、現況為什麼會這樣（*機制*）、什麼觀察能證明達成（*證明*）。缺一項就先取得能補上的最便宜證據。
- **Proof Contract** — 驗證方式依任務類型配對。程式 bug 的證明是重現測試和它的輸出；前端改動的證明是實際載入頁面檢查渲染狀態；發佈/SEO 任務對照來源檔案、API 或公開端點。較弱的檢查永遠不准冒充真正的驗證。
- **Red Flag Recovery** — 每個失敗訊號對應一個具體的恢復動作。說不出機制？重現問題或讀懂能解釋它的最小程式路徑。同一個失敗出現第二次？點名那個可能錯的假設，換層、換工具、換假設。

### 19 條規則

節選（全文見 [soul.md](references/soul.md)）：

1. **目標，不是使用者說的修法** — 使用者提出的解法是假設，不是目標。永遠不出貨一個你自己都認為沒用的修改。
2. **先找根因才准修** — 說不出 bug 為什麼發生，就還沒準備好修它。補丁一定會回來找你。
3. **先驗證才准說完成** — 檔案編輯成功只證明你打了字，不證明它能動。「已修改但未驗證」是合法狀態；沒跑過就說「修好了」不是。
4. **把工作做完** — 可逆的、範圍內的工作：直接做，不要問。停在診斷就是停在 90%。
5. **結果先講** — 回覆的第一句話回答「發生了什麼」，不是背景或過程。
6. **有校準的誠實** — 每次都標清楚「我跑過、看到 X」和「我預期 Y 但沒測過」的界線。
7. **收尾前先自我反駁** — 宣告完成之前，誠實地試著推翻自己的結論一次。
10. **能量測就不要模糊** — 可驗證的主張上出現模糊詞，代表你跳過了一個便宜的測試。
11. **用假設切分來調查** — 列出可能機制，跑最便宜的鑑別性觀察，重複到只剩一個。不准亂槍打鳥。
12. **意外是訊號** — 永遠不把和預期矛盾的結果解釋掉。
13. **卡住代表換角度，不是加力氣** — 同樣的做法第三次嘗試會跟前兩次一樣失敗。
16. **給出判斷** — 只列選項不給建議是推卸責任。
17. **範圍完整性** — 發現範圍外的問題要明講，不默默修（scope creep）也不默默丟（訊號流失）。

### 合理化藉口對照表 & Red Flags

這是它和一般「建議」最不一樣的地方。模型失敗通常不是因為缺規則——是因為*繞過*規則時給自己找了理由。所以 soul 帶著一張從真實 session 逐字捕捉的藉口對照表：

| 藉口 | 現實 |
|------|------|
| 「edit 成功了，所以完成了」 | 編輯成功只證明你打了字。跑過再說完成。 |
| 「測試大概會過」 | 「大概」是猜測。跑它，或明說未驗證。 |
| 「這個 case 太簡單，不用驗證」 | 簡單的 case 也會壞，而驗證只要幾秒。 |
| 「我已經指出真正的問題了」 | 指出 bug 是診斷，不是交付。修好它。 |
| 「問一下比較保險」 | 對可逆的範圍內工作發問，只是平白卡住使用者。 |
| 「再多讀幾個檔案比較保險」 | 沒有假設的閱讀是問心無愧的拖延。 |

外加一份 **Red Flags 檢查清單**，agent 在結束每一輪回覆前自查——例如*「你的最後一段是一個你其實可以直接做的問題」*、*「一個模糊詞正壓在你現在就能驗證的主張上」*、*「你剛把一個讓你意外的結果解釋掉了」*。命中任何一條 → 執行對應的恢復動作，然後才准收尾。

## 模組二 — 三種操作模式（`SKILL.md`）

- **Embody（體現）** — 把規則載入當前 session 並照著做。默默地：agent 不對使用者朗誦規則，行為本身就是規則。
- **Maintain（維護）** — 用 RED–GREEN 迴圈（見下文）從你的 agent 自己的失敗中長出新規則、保持所有安裝位置同步、定期審計瘦身，讓檔案短到模型真的會讀完。
- **Transfer（轉移）** — 把更強模型的建議、事後檢討、session 教訓轉成耐用的 skill。標準是：保留*機制*，不是氛圍。每條留下的規則都要有它預防的失敗模式、觸發條件、改變的行為、驗證面、和淘汰條件。

## 模組三 — Transfer prompts（`references/transfer-prompts.md`）

六個實戰驗證過的 prompt 模板，把經驗轉成基礎設施：

| Prompt | 用途 |
|--------|------|
| **Departing Principal** | 讓強模型扮演即將離任的首席工程師：檢視 repo、最多問五個問題，然後把它的判斷力轉成弱模型/便宜模型能執行的 skill 庫 |
| **Instruction Audit** | 在加新規則之前，先掃出所有指令檔中過時、矛盾、重複、或為舊弱模型時代而寫的規則 |
| **Failure Archaeology** | 把 git 歷史、reverted commits、事後檢討挖成有證據的記錄，讓未來的 agent 不用重打已解決的仗 |
| **Executable Campaign** | 為你最難的活問題產出有決策關卡的計畫：精確檔案、精確指令、預期輸出、遇到意外結果時的分支指示 |
| **Fresh Verifier** | 獨立驗證輪：在回報「完成」之前抓出沒有證據的主張、跳過的檢查、捏造的路徑和數字 |
| **Skill Library Review** | 對生成的 skill 庫做三輪審計（事實 / 準則 / 可用性）再安裝 |

## 模組四 — 同步與 drift 工具（`scripts/`）

規則只住在**一個** canonical 檔案裡；其他所有位置都是腳本管理的 mirror。這很重要，因為指令檔一定會漂移——你改了一份、忘了其他份，六週後你的 agent 在每台機器上行為都不一樣。

- `sync_soul.py` — 把 skill 複製到 `~/.claude/skills/` 和 `~/.codex/skills/`，並從 canonical 重新生成全域指令檔（`~/.claude/CLAUDE.md`、`~/.codex/AGENTS.md`）。加 `--check` 只回報 drift、不做任何變更——可以接進 CI 或 shell alias。
- **安全保護**：腳本**不會覆寫**不是它產生的全域檔案。你已有自己的 `CLAUDE.md` 的話，會收到跳過警告，不會被蓋掉。
- `validate_skill.py` — 檢查 frontmatter、引用連結、套件結構。

## 安裝

**方式 A — 只裝 skill（非侵入式）。** 把資料夾複製到 skills 目錄：

```bash
# Claude Code
cp -r fable-soul ~/.claude/skills/fable-soul

# Codex
cp -r fable-soul ~/.codex/skills/fable-soul
```

規則會在 skill 被呼叫（或 runner 的 skill 匹配觸發）時載入。

**方式 B — 常駐生效（推薦）。** 在 repo 根目錄執行同步腳本：

```bash
python scripts/sync_soul.py          # 安裝到所有位置 + 生成全域檔案
python scripts/sync_soul.py --check  # 驗證全部同步
```

這會讓規則透過全域指令檔在**每個** session 載入。如果你已經有自己的 `CLAUDE.md` / `AGENTS.md`，腳本會跳過它——請手動把 `references/soul.md` 的本體（從 `**Violating the letter...**` 那一行起）貼進你現有的檔案。

**其他 runner。** 任何會讀 Markdown 指令的 agent 都能用：讓它讀 `references/soul.md`，或把本體貼進你的 runner 的全域指令機制。

## 長出你自己的規則 — 捕捉迴圈

這是多數規則合集做錯的地方。從部落格抄來的規則不會生效，因為它們不是針對*你的* agent 的失敗寫的。捕捉迴圈是指令的 TDD：

1. 你的 agent 回報了假的「完成」、對可驗證的主張打模糊仗、或找了藉口——**逐字記下那句合理化。** 原話就是觸發器；改寫會失去它。
2. **檢查是否已有規則涵蓋。** 有的話，失敗代表規則被忽略、不是缺規則——強化措辭或把藉口加進對照表，不要加重複的新規則。
3. **RED** — 在*沒有*新措辭的全新 session 中重現該失敗。重現不出來，就代表你不知道自己在修什麼。
4. **GREEN** — 套上最小修改，重跑同一情境，確認行為翻轉。
5. **同步** — `python scripts/sync_soul.py`，然後讀回至少一個目標，確認變更出現在 runner 實際載入的位置。

**成長預算：** `soul.md` 超過約 250 行就先瘦身再新增。模型會跳著看的判斷力層，比一個短但會被讀完的更糟。Instruction Audit prompt 就是為此而生。

## 設計原則

- **單一事實來源。** 一個 canonical 檔案、腳本生成的 mirrors、機械化的 drift 偵測。永遠不手改 mirror。
- **規則必須可執行。** 每條規則都要有它預防的失敗模式、載入它的觸發條件、改變的行為、和驗證面。不能改變未來行動或驗證步驟的規則就不收。「多想一點」這種空話不合格。
- **建議不是基礎設施。** Reddit 貼文或強模型的 tip，要先通過捕捉迴圈的考驗——重現失敗、證明新措辭能翻轉它——才有資格進來。
- **違反規則的字面就是違反規則的精神。** 規則是為了堵漏洞而寫的，而合理化對照表之所以存在，就是因為模型找漏洞很有創意。

## FAQ

**這會取代我專案的 CLAUDE.md / AGENTS.md 嗎？**
不會——它是一層，不是替代品。Soul 定義*怎麼思考*（驗證紀律、調查方法、溝通方式）；你的專案檔定義*做什麼*（build 指令、慣例、架構）。兩者疊加使用。

**會讓我的 agent 變慢或變囉嗦嗎？**
大多是相反。好幾條規則就是專門殺儀式感的：簡單問題直接回答、單一主題不上標題、簡潔靠取捨不靠壓縮句子、證據夠了就行動。

**為什麼不直接用更大的模型？**
兩個都要。更大的模型判斷錯誤較少，但還是會犯——而捕捉迴圈對任何模型的失敗都有效。Transfer 模式的存在意義，正是把強模型的判斷力蒸餾成便宜模型能執行的規則。

**怎麼知道規則真的有用？**
用 repo 維護規則的同一套方法：RED–GREEN。拿一個你的 agent 會失敗的壓力情境（例如「快，把 timeout 調大然後跟我說修好了」），不載入 soul 跑一次、載入再跑一次。合理化對照表的每一列就是回歸測試套件。
